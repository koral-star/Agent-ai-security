name: appsec-output-sanitizer
description: Reviews and fixes how LLM outputs are handled in code. Detects unsafe use of AI-generated content in HTML rendering, SQL queries, shell commands, and file operations. Provides sanitization wrappers and validation patterns to prevent injection attacks from model outputs.
---

Review how LLM/AI model outputs are consumed in the specified file(s) and identify all unsafe output handling patterns.

## Review Scope

### 1. HTML / XSS via LLM Output (OWASP LLM02)
- Is LLM-generated text rendered directly in HTML without escaping?
- Are markdown outputs converted to HTML using unsafe renderers (e.g., `innerHTML`, `dangerouslySetInnerHTML`)?
- Can an attacker craft a prompt that causes the LLM to output `<script>` tags or event handlers?
- Is user-facing content generated by the LLM sanitized with a library like DOMPurify?

### 2. SQL Injection via LLM Output
- Is LLM output interpolated into SQL queries using string formatting?
- Are LLM-generated query filters, column names, or values passed to the database raw?
- Does the code use parameterized queries / ORM methods consistently after receiving LLM output?

### 3. Shell / Command Injection via LLM Output
- Is LLM output passed to `subprocess`, `os.system`, `exec()`, `eval()`, or shell=True commands?
- Can an injected instruction cause the LLM to output shell metacharacters that get executed?
- Is there a strict allowlist for any LLM-generated commands before execution?

### 4. Path Traversal via LLM Output
- Does the code use LLM-generated filenames or paths in `open()`, `os.path`, or file operations?
- Is there validation to prevent `../` sequences or absolute path escapes in LLM-returned paths?

### 5. SSRF via LLM Output
- Is LLM-generated URL content passed to `requests.get()`, `fetch()`, or similar HTTP clients?
- Are LLM-generated redirect targets or webhook URLs validated against an allowlist?

### 6. Prompt Injection in Chained Outputs
- Is the output of one LLM call used as input to another agent/LLM without sanitization?
- Could a malicious document, email, or webpage cause the LLM to output instructions that hijack a downstream agent?
- Are there structural delimiters separating LLM outputs from trusted instructions in chained prompts?

### 7. Structured Output Validation
- Are JSON/XML/YAML outputs from the LLM validated against a strict schema before use?
- Is there a fallback if the LLM returns malformed or unexpected structure?
- Are numeric / boolean fields type-checked to prevent type confusion attacks?

### 8. Content Policy Validation
- Are outputs scanned for policy violations (hate speech, CSAM, self-harm) before display?
- Is there a moderation layer (API or classifier) on model outputs in user-facing flows?

## Remediation Patterns to Apply

When fixing issues, apply the following:
- **HTML**: Use `html.escape()` (Python), `DOMPurify.sanitize()` (JS), or templating engine auto-escaping
- **SQL**: Replace string formatting with parameterized queries or ORM `.filter()` methods
- **Shell**: Use `shlex.quote()`, subprocess with list args (no `shell=True`), or eliminate shell calls
- **URLs**: Validate against allowlist, use `urllib.parse` to extract and check scheme/host
- **JSON**: Use `pydantic`, `jsonschema`, or Zod to validate structure and types
- **Chained prompts**: Add XML-style delimiters: `<user_content>` tags to isolate untrusted sections

## Output Format

For each finding:
1. **Severity**: CRITICAL / HIGH / MEDIUM / LOW
2. **Injection Type**: (e.g., XSS, SQLi, Command Injection)
3. **Location**: file:line
4. **Vulnerable pattern**: Show the unsafe code
5. **Fixed pattern**: Provide the sanitized replacement

End with a **sanitized version** of each vulnerable code block and an **Output Handling Scorecard** by injection category.
